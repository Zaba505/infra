Choose the datastore(s) that hold tenant state for the self-hosted-application-platform capability.

- TR-07: All inter-service communication must traverse the Cloudflare -> GCP path.

- Access control / query scoping

# --- Issue 2: Tenant data partitioning model ---
gh issue create \
  --title "ADR: Tenant data partitioning model (self-hosted-application-platform)" \
  --label "adr,capability:self-hosted-application-platform" \
  --body "$(cat <<'EOF'
## Decision to make
Choose how tenant data is partitioned within the chosen storage system: shared-store-with-tenant-id, database-per-tenant, project-per-tenant, or hybrid.

## Forcing technical requirements
- TR-01: Tenants must be isolated such that no tenant can read another's state.
- TR-03: Tenant-facing observability data must be queryable per-tenant within their data scope only.

## Depends on
- ADR: Tenant state storage system

## Out of scope
- Choice of storage system itself
- Access control mechanism
- Backup/export/import
EOF
)"

# --- Issue 4: Tenant data backup, export, and import ---
gh issue create \
  --title "ADR: Tenant data backup, export, and import (self-hosted-application-platform)" \
  --label "adr,capability:self-hosted-application-platform" \
  --body "$(cat <<'EOF'
## Decision to make
Choose backup cadence, retention, and restore SLO; the eviction-export mechanism that lets a tenant take their data with them; and the idempotent import mechanism for new tenants.

## Forcing technical requirements
- TR-05: A tenant evicted from the platform must be able to take their data with them.
- TR-06: New tenants must be able to migrate existing data into the platform without loss or corruption.

## Depends on
- ADR: Tenant state storage system
- ADR: Tenant data partitioning model

## Out of scope
- Storage system, partitioning, access control.
EOF
)"
