  --body "$(cat <<'EOF'
### Decision

Where does a tenant's business state physically live? This ADR picks the datastore (or class of datastore) and the account/project boundary in which it sits. It does not decide partitioning within the store (see sibling ADR) or the compute-layer isolation boundary (see sibling ADR).

### Technical requirements addressed

- [TR-01](../tech-requirements.md#tr-01-tenants-must-be-isolated-such-that-no-tenant-can-read-anothers-state) — tenant isolation at the data layer
- [TR-04](../tech-requirements.md#tr-04-operator-initiated-tenant-updates-must-complete-without-tenant-perceived-downtime-for-online-workloads) — state must remain reachable through no-downtime updates
- [TR-05](../tech-requirements.md#tr-05-a-tenant-evicted-from-the-platform-must-be-able-to-take-their-data-with-them) — state must be exportable on eviction
- [TR-06](../tech-requirements.md#tr-06-new-tenants-must-be-able-to-migrate-existing-data-into-the-platform-without-loss-or-corruption) — store must accept idempotent imports

### Parent capability

[self-hosted-application-platform](../docs/content/capabilities/self-hosted-application-platform/_index.md)

  --body "$(cat <<'EOF'
### Decision

Given the storage choice from the sibling ADR, how is tenant data keyed, partitioned, or segregated within it? - [TR-03](../tech-requirements.md#tr-03-tenant-facing-observability-data-must-be-queryable-per-tenant-within-their-data-scope-only) — per-tenant query scoping reuses this partitioning model
